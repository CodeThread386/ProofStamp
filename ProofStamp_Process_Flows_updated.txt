Process 1: User Registration and Proof Passport Creation
This process covers how a new user creates an account and establishes their Proof Passport — their verified identity on the platform that links all of their stamped files together.
Step-by-Step Flow
User Lands on ProofStamp  —  User visits proofstamp.io for the first time.
Sign Up  —  User clicks Sign Up (Google OAuth) and completes Aadhaar eKYC via DigiLocker / Setu API for a legally binding identity. 
Email Verification  —  System sends a verification email. User clicks the link to confirm their address. Account is inactive until verified.
Profile Setup  —  User fills in their Proof Passport details:
Display name (e.g. Sarthak Trivedi)
Username handle (e.g. @sarthak) — becomes their public URL
Cryptographic Keypair Generation  —  System automatically generates an RSA keypair for the user:
Public key stored in the database and embedded in all of the user's stamps
Private key used to cryptographically sign stamps, stored securely on the backend
Proof Passport Created  —  System creates the user's Proof Passport record in the database (Identity verified via Aadhaar eKYC, ensuring rigorous BSA 2023 Section 63 compliance):
Passport ID assigned (e.g. PP-2026-00192)
Username registered and reserved
Public key stored
Creation timestamp recorded
Verified status set to true after email confirmation
Public Profile Goes Live  —  The user's public Proof Passport page is now accessible at proofstamp.io/@username. It shows their identity, stamped files, and provenance chains.
User Reaches Dashboard  —  User lands on their personal dashboard showing: total files stamped, provenance chains, and a shortcut to stamp a new file.
Data Created in This Process
User account record (id, email, password hash, created_at)
Passport record (passport_id, user_id, username, display_name, bio, public_key, links, created_at)
Process 2: File Stamping
This is the core process of ProofStamp. The creator uploads a file and the system embeds cryptographic proof of ownership directly into the file using both metadata injection and steganographic watermarking. The creator gets back a stamped version of the file that carries proof wherever it travels.
Step-by-Step Flow
User Navigates to Stamp Page  —  Logged-in user clicks Stamp a File on the dashboard or navigates to /stamp.
File Upload  —  User uploads their file. Supported formats include images (JPG, PNG, WebP), videos (MP4, MOV), audio (MP3, WAV), and documents (PDF).
Client-Side Hashing  —  Immediately after upload, the browser computes a SHA-256 hash of the raw file. This hash is the file's unique fingerprint. The file is not sent to the server yet.
Note: Hashing on the client side means the original file never leaves the user's device before proof is established. This is important for privacy and trust.
User Fills Stamp Metadata  —  User fills in details about the file:
Title — a human-readable name for the file
Description — optional context about the work
License type — All rights reserved, Creative Commons (CC-BY, CC-BY-SA), or Public Domain
Is this a derivative? — toggle that triggers the provenance chain flow if the file is based on someone else's stamped work
User Clicks Stamp This File  —  User confirms and submits. File and metadata are sent to the backend.
Backend: Dual Embedding Pipeline  —  Two embedding operations happen in parallel on the backend:
Track A — Metadata Injection (C2PA)
System injects a structured metadata block using C2PA media provenance standards into the file's existing metadata layer 
Metadata contains: passport_id, stamp_id, timestamp, SHA-256 hash of the original, creator username, and a digital signature
Digital signature is created by signing the stamp record with the creator's private key
This metadata is readable by any tool that can inspect file metadata
Track B — Steganographic Watermark
System runs LSB (Least Significant Bit) steganography on the file
The stamp_id is embedded invisibly into the pixel data (for images and video) or audio sample data
This watermark is imperceptible to the human eye or ear
It survives common transformations: reposts, screenshots, recompression, color filters, and format conversion
For documents, the stamp is embedded into the binary layer of the file
Database Record Created  —  A stamp record is saved to the database (Data Retention and Key Archival SLA policies enforced):
stamp_id — unique identifier for this stamp
passport_id — links to the creator's Proof Passport
original_hash — SHA-256 of the file before stamping
stamped_hash — SHA-256 of the file after stamping (these differ slightly due to embedding)
title, description, license — user-provided metadata
file_type — image, video, audio, or document
thumbnail_url — stored preview image
parent_stamp_id — null for original works, populated for derivatives
signature — cryptographic signature of the stamp
created_at — timestamp
Output Delivered to User  —  User receives:
The stamped file — downloadable, with proof embedded inside
Verification link — proofstamp.io/v/stamp_id
QR code — scannable link to the verification page
ProofStamp Certificate — a downloadable PDF certificate, complying with BSA Section 63 Part B, featuring an Aadhaar-based Expert Signature applied via Setu eSign API.
Passport Profile Updated  —  The new stamp appears on the creator's public Proof Passport page at proofstamp.io/@username.
Data Created in This Process
Stamp record in the database (all fields listed in Step 7)
Stamped file stored in cloud storage (Cloudinary or S3)
Certificate PDF generated and stored
Process 3: File Verification
Anyone — not just registered users — can verify a file. This process determines whether a file carries a valid ProofStamp, whether it has been tampered with since it was stamped, and who the original creator is.
Step-by-Step Flow
Verifier Navigates to Verify Page  —  Anyone visits proofstamp.io/verify. No login required.
Entry Point — Two Options  —  The verifier has two ways to begin verification:
Option A — Upload the File
Verifier uploads the file they want to verify
Browser extracts the metadata layer and reads the stamp_id from it
Browser also runs steganographic extraction to read the stamp_id from the pixel or audio data
Both extracted stamp IDs are sent to the backend for lookup
Option B — Paste Verification Link or Scan QR Code
Verifier pastes the verification URL (proofstamp.io/v/stamp_id) or scans the QR code
stamp_id is extracted directly from the URL
Sent to backend for direct database lookup
Backend Hash Computation  —  Backend recomputes the SHA-256 hash of the uploaded file and compares it to the original_hash stored in the database at time of stamping.
Database Lookup  —  Backend queries the stamps table for the stamp_id. Three outcomes are possible:
Outcome A — Authentic
Condition: Stamp found in database AND recomputed hash matches the stored original_hash exactly.
System displays:
Green verified badge: Authentic
Owner's Proof Passport card (avatar, name, username, profile link)
Original timestamp of stamping
File title, description, and license
Full provenance chain (if the file is part of one)
Downloadable ProofStamp Certificate
Verification badge image the owner can embed on their website
Outcome B — Tampered
Condition: Stamp found in database BUT recomputed hash does NOT match the stored original_hash.
System displays:
Red tamper alert: This file has been modified since it was stamped
Original owner's Proof Passport card (who it really belongs to)
Original timestamp vs current file date
Side-by-side visual: stored original thumbnail vs currently uploaded file
AI-generated heatmap highlighting regions that were altered
Tamper severity rating: Minor edit, Significant edit, or Major manipulation
Outcome C — Unregistered
Condition: No stamp_id found in metadata or steganographic layer, or stamp_id not found in database.
System displays:
Grey notice: No ProofStamp found on this file
Prompt: If this file is yours, stamp it now
Prompt: If you found this file being misused, report it
Process 4: Provenance Chain — Edit Lineage
The provenance chain allows collaborators, editors, and publishers to create derivative works that are cryptographically linked to the original. Every version of a file in the chain points back to the original creator. This creates an immutable, verifiable edit history.
Part A: Creating a Derivative Work
Editor Has a Stamped File  —  An editor or collaborator has a legitimately obtained file that already carries a ProofStamp from the original creator.
Editor Makes Their Changes  —  Editor modifies the file — for example: cropping an image, color grading a photo, translating a document, or remixing audio.
Editor Logs In and Goes to Stamp Page  —  Editor navigates to /stamp and uploads their modified version of the file.
Editor Toggles This is a Derivative Work  —  Editor enables the derivative toggle in the stamp metadata form.
System Asks for the Original  —  System prompts: Link to the original stamped work. Editor either:
Pastes the original verification URL (proofstamp.io/v/original_stamp_id), or
Uploads the original stamped file so the system can extract the stamp_id
System Verifies the Original Stamp  —  Backend looks up the parent_stamp_id and confirms it exists. The original creator's Proof Passport is retrieved.
Editor Describes the Edit  —  Editor fills in what they changed:
Edit description: Color graded for publication
Edit type: Crop, Color grade, Translation, Remix, Format conversion
New Stamp Created as Child Node  —  System creates a new stamp record linked to the parent:
stamp_id: new unique ID
parent_stamp_id: the original stamp's ID
editor_passport_id: the editor's Proof Passport
original_creator_passport_id: always preserved from the root of the chain
edit_description: user-provided description
timestamp: now
Provenance Chain Updated  —  The chain is now extended. Anyone verifying any file in the chain will see the complete lineage.
Original Creator Notified  —  The original creator receives a notification: Your work has a new derivative by @editor.
Part B: Viewing a Provenance Chain
Anyone Verifies a File in the Chain  —  Any file in the chain — original or derivative — can be uploaded to ProofStamp or looked up via link.
System Resolves the Full Chain  —  Backend performs a recursive query on the stamps table using parent_stamp_id links to traverse the full lineage from root to the current node.
Chain Displayed Visually  —  The full chain is displayed as a tree:
Each node shows: who stamped it, what edit was made, and when
Each node links to that person's Proof Passport
The root node (original creator) is always shown at the top and highlighted
The currently verified file is highlighted in the tree
Creator Can See All Derivatives  —  From their Proof Passport, the original creator can see how many derivative versions exist, who made them, and when.
Example Provenance Chain
The following is an example of what a complete provenance chain looks like:
@photographer  —  Original RAW photo  —  May 1, 2026  (ROOT)
@editor  —  Color graded and cropped  —  May 3, 2026
@publisher  —  Resized for web and watermarked  —  May 5, 2026
Any person verifying the publisher's version will see all three nodes above, with @photographer always shown as the original creator.
Process 5: Public Proof Passport Page
Every registered user has a public Proof Passport page. This is their verified creator identity on the platform. It functions like a portfolio — but every item in it is cryptographically proven to be theirs.
Step-by-Step Flow
User or Visitor Navigates to Passport Page  —  Anyone visits proofstamp.io/@username — no login required.
Page Loads Creator Identity  —  The page displays:
Creator avatar, display name, and @username
Bio and external profile links (LinkedIn, GitHub, Twitter)
Member since date
Stats: total files stamped, total provenance chains created, total derivatives credited
Stamped Files Section  —  A gallery of all files this creator has stamped, showing:
File thumbnail
Title and file type
Date stamped
License type
Number of derivatives in the chain (if any)
Each file is clickable and opens its full verification page
Provenance Chains Section  —  A list of all provenance chains rooted in this creator's work:
Shows chain depth (how many derivatives exist)
Shows all contributors in the chain
Clicking a chain opens the full visual lineage tree
Verification Badge  —  The page displays a Verified by ProofStamp badge. Creators can embed this badge on their personal websites or portfolios, linking back to their Proof Passport.
Share Options  —  Creator can share their Proof Passport URL via social media, include it in their bio, or add it to their email signature as a link to their verified portfolio.
Database Schema
The following tables support all five processes described in this document.
USERS Table
id — unique user identifier (primary key)
email — user's email address (unique)
password_hash — bcrypt hashed password
created_at — account creation timestamp
PASSPORTS Table
id — unique passport identifier (primary key, e.g. PP-2026-00192)
user_id — foreign key to USERS
username — unique @handle
display_name — full name shown on profile
bio — optional profile description
public_key — RSA public key for signature verification
links — JSON object of social/external profile links
created_at — passport creation timestamp
STAMPS Table
id — unique stamp identifier (primary key)
passport_id — foreign key to PASSPORTS (creator)
original_hash — SHA-256 of the file before stamping
stamped_hash — SHA-256 of the file after embedding
title — user-provided title
description — user-provided description
license — license type string
file_type — image, video, audio, or document
thumbnail_url — URL to stored preview image
file_url — URL to the full stamped file in cloud storage
parent_stamp_id — foreign key to STAMPS (null for original works)
original_creator_passport_id — always points to root creator even in deep chains
edit_description — description of changes (for derivatives)
signature — cryptographic signature of the stamp record
created_at — stamping timestamp
Technology Stack
Frontend
React with Tailwind CSS for UI components and styling
Framer Motion for animations and transitions
Web Crypto API for client-side SHA-256 hashing
QR code generation library for verification QR codes
Backend
Node.js with Express for API server
Python microservice for steganography (stegano library for LSB embedding and extraction)
crypto module for RSA signing and verificationAutomated legal notice systems and robust circuit-breaker patterns for reliable eKYC
pdf-lib for ProofStamp Certificate generation
Database
PostgreSQL — supports recursive queries needed for provenance chain traversal via parent_stamp_id
File Storage
Cloudinary or AWS S3 for storing stamped files and thumbnails
Authentication & Legal Infrastructure
JWT tokens for session managementSetu eSign API and DigiLocker for Aadhaar eKYC and Expert Signatures
bcrypt for password hashing
Hosting
Vercel for the React frontend
Railway for the Node.js backend and Python microservice

Demo Script for Judges
Show a real photo  —  Upload it to ProofStamp. Stamp it. Show the certificate and QR code.
Open verification page  —  Upload the same photo. Show the green Authentic result with the owner passport card.
Tamper the file  —  Open the photo in any editor, make one small change, save it.
Verify the tampered file  —  Upload it. Show the red Tampered result with the heatmap of what changed.
Show the provenance chain  —  Create a derivative, link it to the original, show the full lineage tree.
Total demo time: approximately 3 minutes. Every step is visceral and immediately understandable to non-technical judges.
